Documentation : Algorithmes de Peterson et Dekker

1. Introduction à la Concurrence et à l'Exclusion Mutuelle

Contexte :

  • Concurrence : Dans les systèmes informatiques modernes, plusieurs programmes ou tâches sont souvent exécutés en parallèle, soit via un système multi-tâches, soit sur des multi-processeurs.
  • Problème : Quand deux processus accèdent simultanément à la même ressource, des interférences peuvent se produire, risquant de corrompre les données.

Contenu :

  • Exclusion mutuelle : Garantir qu’un seul processus accède à une ressource partagée à la fois, pour éviter des conflits.
  • Objectif des algorithmes de synchronisation : Contrôler l'accès aux ressources partagées de façon ordonnée, pour éviter les conflits.

2. Présentation de l'Algorithme de Dekker

Contexte : L'algorithme de Dekker, proposé par Theodorus Dekker dans les années 1960, est l'une des premières solutions d'exclusion mutuelle.

Contenu :

  • Principe de fonctionnement :
    • Dekker utilise deux variables booléennes (ou flags) pour signaler l’intention d’entrer dans la section critique, et une variable tour pour gérer les conflits.
  • Étapes de l'algorithme :
    • Chaque processus active son flag pour signaler son intention.
    • Si les deux processus activent leur flag simultanément, le tour décide qui entre en premier.
  • Caractéristiques :
    • Assure l'exclusion mutuelle et la vivacité.

3. Présentation de l'Algorithme de Peterson

Contexte : L'algorithme de Peterson, développé par Gary Peterson dans les années 1980, améliore la logique de Dekker en simplifiant son application et en garantissant les mêmes résultats.

Contenu :

  • Principe de fonctionnement :
    • Peterson utilise un tableau de flags pour signaler l’intention des processus, ainsi qu'une variable tour pour déterminer la priorité.
  • Étapes de l'algorithme :
    • Un processus met son flag à vrai et cède le tour à l'autre.
  • Caractéristiques :
    • Assure l'exclusion mutuelle et l'absence de famine.

4. Comparaison entre Dekker et Peterson

Contexte : Cette section explique pourquoi Peterson est souvent préféré et ses différences clés avec Dekker.

Contenu :

  • Simplicité : Peterson est plus facile à comprendre et à mettre en œuvre.
  • Correction formelle : Plus facile à prouver correct grâce à sa simplicité.

5. Exemples en Code

Contexte : Ces exemples de code illustrent le fonctionnement de chaque algorithme.

Algorithme de Dekker :

flag = [False, False]
tour = 0

def enter_critical_section(i):
    flag[i] = True
    while flag[1 - i]:
        if tour != i:
            flag[i] = False
            while tour != i:
                pass
        flag[i] = True

Algorithme de Peterson :

flag = [False, False]
tour = 0

def enter_critical_section(i):
    flag[i] = True
    tour = 1 - i
    while flag[1 - i] and tour == 1 - i:
        pass

6. Analyse des Propriétés

Les deux algorithmes garantissent l'exclusion mutuelle et l'équité. Cependant, Peterson est plus facile à prouver correct en raison de sa simplicité.

7. Limites et Alternatives Modernes

Ces algorithmes sont conçus pour deux processus. Pour des systèmes multi-processus modernes, on utilise des sémaphores, mutex, ou spinlocks, qui sont mieux adaptés aux environnements complexes.

8. Conclusion

Les algorithmes de Peterson et Dekker sont des solutions pionnières pour la synchronisation concurrente, mais sont désormais limités aux environnements simples. Ils restent essentiels pour comprendre la synchronisation en informatique.
